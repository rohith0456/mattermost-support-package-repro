# Usage Guide

This guide covers all `mm-repro` CLI commands, common workflows for different issue categories, and troubleshooting tips.

---

## Prerequisites

Before running any command, ensure:

1. **Docker Desktop is running** — all generated environments use Docker Compose
2. **Go 1.22+** is installed (if building from source)
3. **Sufficient disk space** — a full-stack repro with OpenSearch can use 5–10 GB
4. **Ports are available** — the default setup uses 8065, 5432, 9200, 8025

Check all prerequisites at once:

```bash
mm-repro doctor
```

Example output:
```
[ok]  Docker is running (version 26.1.4)
[ok]  docker compose is available
[ok]  Port 8065 is free
[ok]  Port 5432 is free
[ok]  Port 9200 is free
[ok]  Port 8025 is free
[ok]  Disk space: 47 GB available (minimum 10 GB recommended)
[ok]  All checks passed.
```

## Step 1: Get a Support Package

Support packages are ZIP files generated by the Mattermost server. They are typically attached to support tickets in Zendesk or Jira.

**Safe handling reminders:**
- Download from official ticket systems only
- Do not upload to AI tools or untrusted services
- Delete when debugging is complete

---

mm-repro validate --support-package ./customer-support-package.zip

## Step 4: Generate the Environment

```bash
mm-repro init --support-package ./customer-support-package.zip
```

The command will:
1. Parse and redact the support package
2. Resolve the Mattermost Docker image
3. Generate all project files
4. Print a summary of what was created

```
Generating repro environment...
  [1/6] Parsing support package...  done
  [2/6] Redacting secrets (8 values)...  done
  [3/6] Resolving Mattermost image (9.11.2)...  done [exact match]
  [4/6] Resolving plugins (7 plugins)...  done
  [5/6] Generating project files...  done
  [6/6] Writing output...  done

Generated: ./generated-repro/20260309-142301/
  docker-compose.yml
  .env
  Makefile
  README.md
  REPRO_SUMMARY.md
  REDACTION_REPORT.md
  PLUGIN_REPORT.md
  repro-plan.json

Next steps:
  cd ./generated-repro/20260309-142301/
  make run
  open http://localhost:8065
```

### Naming Your Repro

Use `--issue` to embed an issue identifier in the directory name:

```bash
mm-repro init --support-package ./customer.zip --issue "MM-12345-ldap-crash"
# Output: ./generated-repro/MM-12345-ldap-crash-20260309-142301/
```

### Forcing Database Type

If the parser cannot detect the database type, or you want to override it:

```bash
mm-repro init --support-package ./customer.zip --db mysql
mm-repro init --support-package ./customer.zip --db postgres
```

### Specifying Output Directory

```bash
mm-repro init --support-package ./customer.zip --output ./my-repros/ticket-12345
```

## Step 5: Review the Reports

Before starting the environment, read the generated reports:

```bash
cat generated-repro/<timestamp>/REPRO_SUMMARY.md
cat generated-repro/<timestamp>/REDACTION_REPORT.md
cat generated-repro/<timestamp>/PLUGIN_REPORT.md
```

The `REPRO_SUMMARY.md` explains every approximation made. If a critical service was not recreated, you may need to add a flag and re-run `init`.

The `PLUGIN_REPORT.md` lists any plugins marked `[MANUAL]` that you need to install yourself after the environment starts.

---

## Step 6: Start the Environment

```bash
cd generated-repro/<timestamp>/
make run
```

This runs `docker compose up -d` and waits for Mattermost to be healthy.

**Access URLs:**

| Service | URL |
|---------|-----|
| Mattermost | http://localhost:8065 |
| MailHog (email capture) | http://localhost:8025 |
| MinIO console (if enabled) | http://localhost:9001 |
| Grafana (if enabled) | http://localhost:3000 |
| Keycloak admin (if enabled) | http://localhost:8080 |
| OpenSearch Dashboards (if enabled) | http://localhost:5601 |

**Default local admin credentials** (from the generated `.env`):
```
Username: admin_local_repro
Password: Admin_local_repro_1!
```

### Useful Make Targets

```bash
make run        # start all services (docker compose up -d)
make stop       # stop all services (docker compose stop)
make reset      # stop and remove all volumes (clean state)
make logs       # tail all logs
make logs-mm    # tail only Mattermost logs
make ps         # show container status
make shell-mm   # open shell in Mattermost container
make shell-db   # open psql/mysql shell in database container
```

---

## Step 7: Reproduce the Issue

After the environment is running, reproduce the reported issue. Common steps:

- **Configure the same settings** that the customer described (the config is pre-loaded from the support package, minus secrets)
- **Create test users** using the local admin account
- **Install any [MANUAL] plugins** listed in `PLUGIN_REPORT.md`
- **Check MailHog** at http://localhost:8025 for any email-related issues

---

## Step 8: Clean Up

When finished:

```bash
# Stop containers but keep volumes (restart later with make run)
make stop

# Or: remove all containers AND volumes (full clean slate)
make reset

# Then delete the generated directory
cd ..
rm -rf generated-repro/<timestamp>/
```

Never leave generated directories on shared machines or in version-controlled paths.

## CLI Reference (All Commands)

### `mm-repro doctor`

Check prerequisites.

```
Usage: mm-repro doctor [flags]

Flags:
  --json    Output as JSON
  --verbose Show detailed check information
```

### `mm-repro validate`

Validate a support package and show detected signals.

```
Usage: mm-repro validate --support-package <path> [flags]

Flags:
  --support-package <path>   Support package ZIP (required)
  --verbose                  Show all detected config keys
  --json                     Output as JSON
```

### `mm-repro plan`

Preview the repro plan without generating files.

```
Usage: mm-repro plan --support-package <path> [flags]

Flags:
  --support-package <path>   Required
  --with-opensearch          Include OpenSearch in the plan
  --with-ldap                Include LDAP in the plan
  --with-saml                Include Keycloak in the plan
  --with-minio               Include MinIO in the plan
  --with-rtcd                Include RTCD in the plan
  --with-grafana             Include Prometheus + Grafana in the plan
  --db postgres|mysql        Force database type
  --force-single-node        Force single-node topology
  --force-multi-node         Force multi-node topology
  --json                     Output as JSON
```

### `mm-repro init`

Generate a repro environment from a support package.

```
Usage: mm-repro init --support-package <path> [flags]

Flags:
  --support-package <path>   Support package ZIP (required)
  --output <dir>             Output directory (default: ./generated-repro/<timestamp>)
  --issue <name>             Issue identifier for directory naming
  --db postgres|mysql        Force database type
  --force-single-node        Force single-node
  --force-multi-node         Force multi-node
  --with-opensearch          Include OpenSearch
  --with-ldap                Include OpenLDAP
  --with-saml                Include Keycloak (SAML/OIDC)
  --with-minio               Include MinIO (S3)
  --with-rtcd                Include RTCD (Calls)
  --with-grafana             Include Prometheus + Grafana
  --redact-strict            Also redact server addresses and email addresses
  --dry-run                  Print what would be generated without writing files
  --overwrite                Overwrite existing output directory
```

### `mm-repro run`

Start a previously generated repro project.

```
Usage: mm-repro run [flags]

Flags:
  --project <path>   Path to generated repro directory (default: current directory)
  --follow           Follow logs after starting
```

### `mm-repro stop`

Stop a running repro project.

```
Usage: mm-repro stop [flags]

Flags:
  --project <path>   Path to generated repro directory
```

### `mm-repro reset`

Stop and remove all containers and volumes for a repro project.

```
Usage: mm-repro reset [flags]

Flags:
  --project <path>   Path to generated repro directory
  --confirm          Skip confirmation prompt
```

## Workflow Examples by Issue Type

### Cluster / HA Issues

When a ticket involves cluster behavior (e.g., session affinity, plugin state sync, gossip failures):

```bash
# 1. Validate to see if cluster signals are present
mm-repro validate --support-package ./customer.zip

# 2. Generate a multi-node repro (force it if signals are absent)
mm-repro init --support-package ./customer.zip --force-multi-node --issue "MM-99999-cluster-sync"

# 3. Start and observe
cd generated-repro/MM-99999-cluster-sync-*/
make run
make logs

# 4. Check individual node logs
docker compose logs mm-node-1
docker compose logs mm-node-2
docker compose logs mm-node-3
```

See [docs/multi-node.md](multi-node.md) for detailed guidance on HA repros.

### Plugin Issues

When a ticket involves a specific plugin:

```bash
# 1. Check what plugins were detected
mm-repro validate --support-package ./customer.zip

# 2. Generate the environment
mm-repro init --support-package ./customer.zip --issue "MM-88888-jira-plugin"

# 3. Check the plugin report
cat generated-repro/MM-88888-jira-plugin-*/PLUGIN_REPORT.md

# 4. Start and verify plugin installation
cd generated-repro/MM-88888-jira-plugin-*/
make run
make logs-mm | grep -i plugin
```

If the plugin is listed as `[MANUAL]` in `PLUGIN_REPORT.md`, download it from the Mattermost marketplace or the vendor and install it through the System Console after the environment starts.

See [docs/plugin-strategy.md](plugin-strategy.md) for details.

### Authentication Issues (LDAP)

When a ticket involves LDAP login failures, attribute mapping, or group sync:

```bash
# Generate with LDAP
mm-repro init --support-package ./customer.zip --with-ldap --issue "MM-77777-ldap-group-sync"

cd generated-repro/MM-77777-ldap-group-sync-*/
make run

# LDAP is available at ldap://localhost:389
# Admin: cn=admin,dc=repro,dc=local / admin_local_repro_only
# Test users are pre-created:
#   user01 through user10 (password: Password1!)
```

The OpenLDAP container is pre-populated with:
- 10 test users (user01..user10)
- 3 test groups (team-alpha, team-beta, team-leads)
- A structure matching a typical enterprise LDAP tree

Mattermost is configured to use the local OpenLDAP server. LDAP settings (filter, attribute mapping) are copied from the support package (with passwords redacted and server address replaced with the local container).

### Authentication Issues (SAML/OIDC)

When a ticket involves SAML login failures or OIDC configuration:

```bash
# Generate with Keycloak
mm-repro init --support-package ./customer.zip --with-saml --issue "MM-66666-saml-redirect"

cd generated-repro/MM-66666-saml-redirect-*/
make run

# Wait for Keycloak to initialize (can take 2-3 minutes)
make logs | grep -i keycloak

# Keycloak admin console: http://localhost:8080
# Admin credentials: admin / admin_local_repro_only
# Pre-configured realm: mattermost-repro
# Pre-configured users: saml-user01..saml-user05
```

The Keycloak instance is pre-configured with:
- A Mattermost SAML client (metadata auto-imported)
- OIDC client for Mattermost
- Test users with realistic attribute mappings

### Performance / Database Issues

When a ticket involves slow queries, database connection pooling, or timeouts:

```bash
# Generate with Grafana for metrics
mm-repro init --support-package ./customer.zip --with-grafana --issue "MM-55555-slow-queries"

cd generated-repro/MM-55555-slow-queries-*/
make run

# Prometheus: http://localhost:9090
# Grafana: http://localhost:3000 (admin / admin_local_repro_only)
# Pre-loaded dashboards:
#   - Mattermost Overview
#   - Database Connections
#   - Go Runtime
```

For database-specific investigation:
```bash
# Open a database shell
make shell-db

# PostgreSQL
psql -U mmuser_local_repro mattermost_local_repro

# MySQL
mysql -u mmuser_local_repro -p mattermost_local_repro
```

### Search Issues (Elasticsearch/OpenSearch)

When a ticket involves search returning wrong results or indexing failures:

```bash
mm-repro init --support-package ./customer.zip --with-opensearch --issue "MM-44444-search-index"

cd generated-repro/MM-44444-search-index-*/
make run

# OpenSearch: http://localhost:9200
# OpenSearch Dashboards: http://localhost:5601
```

After starting, trigger a re-index from the Mattermost System Console:
`System Console > Elasticsearch > Index Now`

### File Storage Issues (S3/MinIO)

When a ticket involves file upload failures, S3 connectivity, or broken file links:

```bash
mm-repro init --support-package ./customer.zip --with-minio --issue "MM-33333-s3-upload"

cd generated-repro/MM-33333-s3-upload-*/
make run

# MinIO console: http://localhost:9001
# Credentials from .env: MINIO_ROOT_USER / MINIO_ROOT_PASSWORD
```

## Troubleshooting

### "Docker is not running"

```
Error: Docker daemon is not running. Start Docker Desktop and try again.
```

Solution: Start Docker Desktop and wait for it to fully initialize before retrying.

### "Port 8065 is already in use"

```
Error: port 8065 is already in use
```

Solutions:
1. Stop the conflicting process: `lsof -i :8065` then `kill <pid>`
2. Or edit the generated `.env` file:
   ```env
   MM_PORT=8066
   ```
   Then run `make run` again.

### "Port 5432 is already in use"

If you have a local PostgreSQL server running:
```env
POSTGRES_PORT=5433
```

### "Image not found: mattermost/mattermost-enterprise-edition:7.1.0"

Older images may not be on Docker Hub. The tool will fall back to the nearest available version and document the approximation in `REPRO_SUMMARY.md`. You can check available tags at:
https://hub.docker.com/r/mattermost/mattermost-enterprise-edition/tags

### "Support package format unknown"

The parser will attempt best-effort extraction. Run `mm-repro validate --verbose` to see what files were found and which could not be parsed.

### "Mattermost container keeps restarting"

Check the Mattermost logs:
```bash
make logs-mm
# or
docker compose logs mattermost
```

Common causes:
- Database not yet ready (usually resolves automatically within 30s)
- Invalid config value in the generated config
- License validation failure (the generated environment uses no license by default)

### "Keycloak takes a long time to start"

Keycloak can take 2-4 minutes to initialize, especially on first boot. Monitor with:
```bash
docker compose logs -f keycloak
```
Wait for: `Keycloak 24.x.x on JVM (powered by Quarkus ...) started in ...`

### "OpenSearch container exits immediately"

OpenSearch requires `vm.max_map_count` to be set on Linux hosts:
```bash
sudo sysctl -w vm.max_map_count=262144
```

On Docker Desktop (Mac/Windows), this is handled automatically.

### "LDAP sync is failing"

The OpenLDAP stub uses:
- Server: `ldap://ldap:389` (inside Docker network)
- Bind DN: `cn=admin,dc=repro,dc=local`
- Bind Password: from `LDAP_ADMIN_PASSWORD` in `.env`

Check the Mattermost LDAP settings in System Console and compare with the generated config. The `LdapServer` address in the generated config is set to the Docker service name `ldap`, not the customer's original LDAP server address.

### The repro doesn't look like the customer's environment

Remember what cannot be recreated:
- Real customer data, users, and channels
- Exact production performance
- Custom/private plugins (see PLUGIN_REPORT.md for manual steps)
- Exact LDAP/SAML directory contents

Focus on recreating the **configuration** and **topology**, not the data.

## Advanced: Dry Run Mode

Preview what files would be generated without writing anything:

```bash
mm-repro init --support-package ./customer.zip --dry-run
```

This prints the rendered file contents to stdout. Useful for quickly inspecting what the `docker-compose.yml` or `.env` would look like.

## Advanced: JSON Plan Output

Capture the full plan as JSON for scripting or CI use:

```bash
mm-repro plan --support-package ./customer.zip --json | jq '.services[].name'
```

---

## Advanced: Overwriting an Existing Repro

If you need to re-generate to a specific directory (e.g., to add a service):

```bash
mm-repro init --support-package ./customer.zip \
  --output ./generated-repro/my-repro \
  --with-ldap \
  --overwrite
```

Note: `--overwrite` will delete and recreate the entire output directory. Stop the environment first with `make stop` if it's running.
